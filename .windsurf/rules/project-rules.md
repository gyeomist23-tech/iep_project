# Project Rules

## Scope
- 이 저장소의 핵심 목적은 Google News RSS를 이용해 교육 관련 뉴스 브리핑 텍스트 파일을 생성하고, GitHub Actions로 매일 자동 커밋하는 것이다.
- 웹 애플리케이션 기능, DB, 카카오톡 자동 전송 기능은 기본 범위에 포함하지 않는다.

## Output Contract
- 출력 파일 경로는 반드시 `output/YYYY/MM/daily_news_YYYYMMDD.txt` 형식을 유지한다.
- 텍스트 포맷은 다음 규칙을 항상 지킨다.
  - 첫 줄 헤더: `YYYY년 MM월 DD일 디지털 기반 교육 관련 새소식을 전달해드립니다.`
  - 섹션 순서: `[특수교육]`, `[통합교육]`, `[디지털교육]`, `[인공지능]`
  - 빈 섹션은 `(없음)`으로 출력한다.
  - 각 기사 아이템은 정확히 두 줄 `제목 : ...`, `링크 : ...` 형식을 사용한다.

## Classification Rules
- 기사 분류는 제목 우선, 필요 시 summary/description 보조 사용 원칙을 유지한다.
- 중복 키워드 충돌 시 우선순위는 `특수교육 > 통합교육 > 인공지능 > 디지털교육` 이다.
- 기사 채택은 제목 기준 `MUST_HAVE_TERMS` 또는 섹션 키워드 포함 여부를 사용한다.
- 전역 중복 제거 기준은 링크다.
- 기사 발행 시점은 실행 시점 기준 최근 7일 이내만 기본 채택 대상으로 본다.

## Reliability Rules
- RSS/네트워크 오류가 있어도 스크립트는 종료 실패 대신 빈 결과 파일을 생성해야 한다.
- 재시도 로직은 단순하고 예측 가능해야 하며, 과도한 외부 요청을 만들지 않는다.
- GitHub Actions 실패 가능성을 줄이기 위해 Python 표준 라이브러리 중심 구현을 우선한다.

## Change Rules
- 키워드 세트 변경 시 README와 실제 스크립트 동작이 어긋나지 않게 함께 검토한다.
- 출력 포맷 변경은 사용자 복붙 워크플로우에 직접 영향을 주므로 임의 변경하지 않는다.
- Actions 변경 시 `contents: write`, `fetch-depth: 0`, 커밋 메시지 형식 보존 여부를 확인한다.
- 저장소 수정 시 불필요한 파일이나 샘플 산출물은 기본적으로 추가하지 않는다.

## Validation Checklist
- 로컬 실행 명령이 README와 일치하는가
- 결과 파일이 지정 경로에 생성되는가
- 오래된 기사(기본 7일 초과)가 제외되는가
- 빈 결과일 때도 섹션별 `(없음)`이 출력되는가
- 섹션 순서와 총 개수 제한이 요구사항대로 동작하는가
- GitHub Actions가 변경분이 있을 때만 커밋하도록 유지되는가
